1. Whiteboard explanation — You're explaining this paper's mechanism to a smart colleague who hasn't read it. Walk through what they built and how it works.
This work explores the limitations of the recently published last-level branch predictor (LLBM), which was meant to augment an existing baseline TAGE-SC-L design by decoupling high-volume metadata storage from the critical path. However, the authors found a number of limitations hindering the design from achieving its full potential. From this analysis they present an improved version of the design, LLBM-X, which addresses these limitations and more robustly improves performance against a baseline TAGE-SL-L system.

The original LLBM branch predictor is meant to address what the authors dub the 'capacity-latency tradeoff': TAGE predictors become more accurate as storage increases, yet simultaneously become slower due to increased lookup times. To overcome this tension the LLBM implements a larger second-level metadata store decoupled from the critical path. This second-level storage operates by grouping conditional branch histories (or 'patterns') into distinct 'pattern sets'. These pattern sets contain every distinct pattern located along the same global path, where the path is determined by the last 'n' unconditional branches run. So, for example, if the last two unconditional branches are tracked, one pattern set would be all the patterns found before unconditional branch 'a' but after unconditional branch 'b' (note that the LLBM also deliberately ignores the 'm' most recent unconditional branches taken to provide a temporal window during which the LLBP's access latency is hidden). These pattern sets are also referred to as 'contexts'. The LLBP is then able to perform branch predictions by using the same partial pattern-matching algorithm that TAGE implements, but only across the patterns found in the current pattern set. The prediction is then compared to a baseline TAGE - if LLBM's prediction has a longer or equal history length (i.e. the number of past branch outcomes used to predict the direction of the current branch) it overrides TAGE's prediction with its own. The LLBM system also includes the prefetching of upcoming contexts to hide latency, and pattern learning to identify high-confidence contexts.

The authors, by analyzing the performance improvements caused by relaxing various LLBM design constraints (e.g. how does performance improve if we allow an infinite number of patterns-per-context?, etc.) find that the LLBM's performance is most impacted by two issues: poor pattern set utilization and context duplication. Regarding the first point, the authors find that there are a small percentage (14%) of contexts which do not have the capacity to store all useful patterns (i.e. patterns which beat the baseline TAGE prediction), leading to high contention when these contexts are discovered in the program (note that LLBM stores 16 patterns per context). Regarding the second point, the authors find that for short, easy-to-predict branches, the default context depth (i.e. the number of unconditional branches recorded prior to the branch to classify a particular context) leads to unnecessary duplication. This is because the easy-to-predict branches are identifiable from the last couple unconditional branches, while extending the context depth increases the number of separate contexts which all share this same basic pattern. As the authors put it, the more unique unconditional branches exist in contexts, the more times the patterns needed to predict the branch are duplicated.

The authors then come up with a modified version of the LLBM, dubbed the LLBM-X, which addresses these two issues. Essentially, the authors create a classification system within the LLBM: by default, only short contexts exist (which only record the previous two unconditional branches). This addresses the context duplication issue. However, if a context accumulates a significant number of confident patterns, LLBM-X increases the context depth to 64 for that particular context. Note that if branch behavior changes, it is possible for a context to switch back from having a depth of 64 to a depth of two. The authors support this dynamic classification system by adding a new hardware table, dubbed the 'Context Tracking Table' (CTT), which monitors contexts once the number of confident patterns in the context exceeds a certain threshold. Once a context is tracked by the CTT, it is not automatically promoted to a context depth of 64. Rather, its patterns must consistently exceed a threshold history length before the promotion occurs (this is done via a saturating counter). This design also allows for more ranges of history lengths to be tracked, bringing it to parity with the baseline TAGE design in that regard. The authors find that these improvements only increase the design's area by 1.8% and increase accuracy by 3.6% on average.

2. What is the key insight that makes it work? (The "aha" — not what they did, but why it works)
The key insight which makes this design work is twofold. The first aspect is demonstrated in figure 6, which is essentially that number of patterns in a context follows a log-scale, where a small percentage of contexts require a large number of patterns (these being the hard-to-predict branches) while most patterns require less than 16 (these being the easy-to-detect branches). This results in a natural bifurcation of branch types. The second aspect is demonstrated in figure 7, which shows that the number of useful patterns per context correlates to average history length. This allows the LLBM-X hardware to use history length to track if a context requires a higher depth, or whether it can remain in the low-depth region. Taken together, these two findings allow for a clean solution to the problem of LLBM inefficiency.

3. What's the strongest aspect of the evaluation, and what's the weakest? (Methodology critique)
The strongest aspect of the evaluation in my opinion is the thoroughly deconstruction of the causes of LLBM's performance bottlenecks found in Sections III-IV. It does a good job comparing baseline LLBM performance and accuracy with versions in which certain constraints are relaxed, providing a robust foundation for their later design choices. It also covers the breakdown of useful patterns and history lengths per context, as well as the degree to which patterns are duplicated as a function of history lengths. Finally, it shows a good headline result in Figure 9, demonstrating a fairly clean bifurcation in terms of useful predictions between short and long context depth (2 and 64, respectively) as history length increases. This directly informs their design of the LLBM-X prefetcher.

The evaluation of LLBM-X itself is also mostly complete, and has a nice exploration of prefetch efficiency, false path and overriding effects, an ablation study regarding the two optimizations implemented by the LLBM-X (dynamic context depth adaptation and dynamic history range selection), as well as a sensitivity study across a number of parameters. The weakest aspect of the evaluation, however, is the brevity which with the speedup values are presented. Overall, the speedup between LLBM and LLBM-X compared to a baseline TAGE system is quite small (1% improvement vs. 0.71% improvement); obviously its main improvement is in accuracy. However, there are some interesting aspects of the speedup numbers which warrant further investigation, but which are unfortunately never expanded upon. For instance, in the HTTP benchmark LLBM-X outperforms a large (512KB) idealized TAGE-SC-L predictor (despite it being supposed to represent "the theoretical upper bound for comparable storage" according to the authors), while in the Kafka benchmark the baseline LLBP system outperforms LLBM-X and the idealized TAGE.

4. What did the authors not tell you? (Hidden assumptions, missing comparisons, unstated limitations)
As just mentioned, the assumption that the idealized 512KB TAGE system does not appear to always serve as a theoretical upper-bound for speedup. The authors also assert W=2 and W=64 as being the optimal context depths for easy- and hard-to-predict branches, respectively, but with little justification for these values. While the shorter depth (depth=2) value is backed up with some empirical study regarding duplication rates, there does not appear to be a sensitivity study regarding the larger depth value (depth=64).
